# NFT721 Contract Design (Base Sepolia)

## Overview
NFT721 dung de mint NFT duy nhat cho moi noi dung post. Moi post chi duoc phep co toi da 1 token ERC721 tren toan he thong. Contract hien tai da co mint window va whitelist, nhung chua co rang buoc unique theo content.

Muc tieu tai lieu nay:
- Chuan hoa cach mint ERC721 cho post
- Ngan duplicate mint khi nhieu request dong thoi
- Tao event/state ro rang de backend dong bo voi Supabase

## Contract Responsibilities
- Mint 1 ERC721 token cho 1 content key (postId hoac IPFS CID hash)
- Enforce one-content-one-token rule
- Luu metadata URI cho token
- Ho tro backend-controlled minting trong giai doan dau
- Phat event day du de marketplace va database consume

## State Variables
- tokenCounter: bo dem token id tang dan
- mintEndTime: deadline cho mint campaign
- whitelist[address]: dia chi duoc phep goi mint
- contentMinted[bytes32]: danh dau content key da duoc mint chua
- contentToTokenId[bytes32]: map content key -> tokenId
- tokenToContent[tokenId]: map tokenId -> content key
- minterOf[tokenId]: dia chi tao token ban dau

Ghi chu:
- content key nen la keccak256(chainId, postId, creatorWallet, cid)
- key phai deterministic giua frontend/backend

## Core Functions (description only)
- setWhitelist(user, allowed)
  - Owner cap/thu hoi quyen mint caller
- setMintEndTime(newTime)
  - Owner cap nhat cua so mint neu can
- mintUnique(to, tokenURI, contentKey)
  - Validate mint con han
  - Validate caller duoc whitelist
  - Validate contentKey chua duoc mint
  - Mark contentMinted truoc khi mint de tranh race condition
  - Mint token va gan URI
  - Persist map content <-> token
- existsByContent(contentKey)
  - Tra ve true neu content da mint
- tokenIdByContent(contentKey)
  - Tra tokenId theo content de backend truy van nhanh

## Events
- WhitelistUpdated(user, allowed, updatedBy)
- MintWindowUpdated(oldTime, newTime)
- ContentLocked(contentKey, caller)
- UniqueMinted(tokenId, to, contentKey, tokenURI, caller)

Event yeu cau index:
- contentKey va tokenId phai indexed de truy van nhanh

## Access Control
- Owner:
  - Quan ly whitelist
  - Quan ly mint window
- Whitelisted minter (thuong la backend relayer wallet):
  - Duoc mintUnique
- End user wallet:
  - La nguoi nhan token (to)

Khuyen nghi production:
- Dung AccessControl role MINTER_ROLE thay vi whitelist bool neu can mo rong

## Flow Diagrams (text-based)
### Standard Mint Flow
1. Frontend gui yeu cau Mint NFT cho post
2. Backend tao contentKey tu postId + cid + creator
3. Backend check Supabase: post chua co nft721_token_id
4. Backend call existsByContent(contentKey)
5. Neu false, backend goi mintUnique
6. Contract lock contentKey, mint token, emit UniqueMinted
7. Indexer/API worker nghe event, cap nhat bang nfts + posts
8. Backend tao listing tren marketplace

### Race Condition Safe Flow (2 request dong thoi)
1. Request A va B cung tao cung contentKey
2. A vao chain truoc, mintUnique pass, set contentMinted = true
3. B vao sau, require contentMinted == false fail -> revert
4. Backend bat loi revert va tra conflict 409
5. He thong tra ve tokenId hien tai qua tokenIdByContent

## Edge Cases
- Duplicate mint cung content:
  - Xu ly bang mapping contentMinted + check on-chain
- Reorg nhe tren testnet:
  - Backend phai xac nhan N blocks truoc khi finalize DB status
- Mint thanh cong on-chain nhung DB update fail:
  - Dung idempotent event consumer de retry cap nhat
- tokenURI khong hop le:
  - Validate URI format o backend truoc khi broadcast
- mintEndTime het han trong luc pending tx:
  - Tx se revert; backend cap nhat status failed

## Security Considerations
- Khong cho client tu tinh contentKey tuy y; backend phai tao key theo schema co dinh
- Reentrancy khong phai vector chinh khi mint, nhung van nen giu external call toi thieu
- Chong front-run duplicate bang invariant on-chain (khong phu thuoc lock trong DB)
- Owner key phai la multisig hoac hardware wallet trong production
- Event-driven sync phai idempotent dua tren txHash + logIndex
- Contract chi chap nhan network Base Sepolia/testnet da khai bao ro chainId
